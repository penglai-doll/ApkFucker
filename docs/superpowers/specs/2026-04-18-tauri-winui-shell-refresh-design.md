## 1. 背景

当前 Tauri 工作台已经具备案件导入、静态分析、Hook 计划、执行控制、流量证据和报告导出等主流程能力，但界面层仍然偏“功能直出”：

- 顶层壳子几乎没有稳定的视觉层级
- 页面区域之间缺乏明确的导航结构
- 重要状态信息没有集中展示
- 工作台内容长页面平铺，查找与切换成本偏高

用户反馈当前前端“仅仅只是显示，很难用”，因此本次工作不新增分析能力，而是重构前端工作台壳层，使其更接近真实桌面工具，而不是接口集合页。

## 2. 已确认方向

本次重构已确认以下交互与视觉决策：

- 整体风格：`WinUI 风格工作台`
- 导航方式：`左侧带文字的稳定导航`
- 顶部区域：`柔和渐变 + 玻璃感 banner`
- 语言：`全中文`
- 目标：`提升扫读效率、状态感知能力和操作定位速度`

拒绝方案：

- 纯图标轨道式导航：视觉轻，但可发现性不足
- 仅做配色微调：无法解决结构不好用的问题
- 强沉浸暗色分析台：视觉冲击更强，但长期使用压迫感更高

## 3. 设计目标

本次界面优化只做三件事：

1. 把当前 Tauri 前端改造成“像桌面工作台”的稳定壳层
2. 让用户更快找到“当前案件、当前状态、下一步操作”
3. 在不打断现有功能接线的前提下，重排信息层级

非目标：

- 不重写后端 API
- 不改动案件/执行/报告的数据模型
- 不在本次引入新的组件库或 UI 框架
- 不把所有工作台子模块做成完全独立路由页

## 4. 视觉论点

### 4.1 Visual Thesis

一个偏 WinUI 的现代分析工作台：冷静、柔和、清晰，重点依靠层次、间距、磨砂侧栏和顶部状态 banner 建立秩序，而不是依赖堆叠卡片。

### 4.2 Content Plan

- 第一屏：当前模式、当前案件、后端状态、关键动作
- 第二层：当前页面的主操作区
- 第三层：辅助信息、证据、历史和报告

### 4.3 Interaction Thesis

- 左侧导航高亮保持稳定，降低模式切换成本
- 顶部 banner 承担“状态总览 + 快捷动作”而不是纯装饰
- 工作台内部用分区导航和锚点跳转降低长页面浏览成本

## 5. 信息架构

## 5.1 顶层壳层

新的 `AppFrame` 分为三部分：

### 左侧导航栏

职责：

- 承载顶层模式切换
- 显示品牌和当前模式语义
- 为后续扩展设置、环境、帮助等入口预留位置

首版保留两个主入口：

- `案件队列`
- `案件工作台`

设计要求：

- 使用带文字的导航项，不使用纯图标轨道
- 当前项高亮要稳定且强可见
- 导航栏采用磨砂浅色面和轻边界，而非厚重边框

### 顶部 Banner

职责：

- 展示当前模式和关键信息
- 集中展示本地后端健康状态
- 在工作台场景下强调“当前案件正在做什么”

Banner 的显示策略：

- 在 `案件队列`：强调导入、默认工作目录、案件总览
- 在 `案件工作台`：强调当前案件、包名、执行状态与主动作提示

### 主内容区

职责：

- 承载页面主体
- 提供更清晰的 section 间距和宽度约束
- 允许工作台内部继续做次级导航和多区块布局

## 5.2 案件队列页

`CaseQueuePage` 从“表单 + 表格顺序堆叠”调整为三层：

1. 页面头部说明
2. 导入区 + 快速摘要并列
3. 案件列表

具体结构：

- 导入区：样本路径、工作目录、案件标题、主导入按钮
- 快速摘要：案件数、默认工作目录、使用提示
- 列表区：案件表格，强化“进入工作台”的主路径

目标效果：

- 用户进入页面后先看懂“这里是导入和管理案件的地方”
- 不需要滚动就能找到导入操作
- 列表和导入动作不再抢同一层级

## 5.3 案件工作台页

`CaseWorkspacePage` 改成“主 banner + 次级导航 + 分区内容”的结构。

### 工作台头部

展示：

- 当前案件标题
- 包名
- 方法索引状态
- 当前执行状态或最近运行状态
- 快捷动作，例如 `在 JADX 中打开`

### 次级导航

提供工作台内部锚点跳转：

- `静态简报`
- `Hook 工作台`
- `执行控制台`
- `流量证据`
- `证据中心`
- `报告导出`

该导航不额外增加路由，而是作为页内锚点与高可见入口。

### 分区布局

- `静态简报` 与 `执行控制台` 放在首屏主视区
- `Hook 工作台` 独立为大区块，保证编辑、计划和推荐有足够空间
- `流量证据`、`证据中心`、`报告导出` 作为后续区块纵向展开

目标效果：

- 用户进入工作台后，先理解“当前案件状态”
- 再定位“我要看哪一块”
- 最后再进入对应模块深用

## 6. 组件策略

本次不重写所有业务组件，而是采用“壳层重构 + 组件包装增强”：

- `AppFrame`：重写为新的桌面壳层
- `CaseQueuePage`：重排为概览式结构
- `CaseWorkspacePage`：重排为 banner + section 导航 + 分区工作台
- 各个 `Panel` 组件保留原有数据接线，但补充更清晰的 section 语义和样式钩子

这意味着：

- 先解决“结构不好用”
- 不在本次把每个业务子组件都做成大规模拆分

## 7. 样式系统

新增统一的前端全局样式层，负责：

- 颜色变量
- 字体层级
- 栅格和间距
- panel/surface 结构
- 按钮、输入框、表格和状态徽标的基础观感

设计约束：

- 以 CSS 变量驱动配色
- 控制强调色为单一蓝色系
- 允许轻玻璃感和柔和阴影
- 避免深色大背景、过多渐变和营销式大文案

## 8. 风险与边界

### 风险 1：布局重构打断现有功能

应对：

- 不改业务请求与 state 组织方式
- 优先重构页面结构和样式
- 用现有测试回归主路径

### 风险 2：视觉增强导致信息密度下降

应对：

- 队列页保留表格
- 工作台模块仍然使用真实数据组件
- 美化不以牺牲扫描效率为代价

### 风险 3：新增样式造成测试脆弱

应对：

- 测试继续围绕中文文案、导航语义和关键行为
- 不对纯样式细节写过度断言

## 9. 验收标准

满足以下条件视为本次前端优化完成：

- 应用存在稳定的左侧文字导航栏
- 应用顶部存在可识别的状态 banner
- `案件队列` 首屏可直接完成导入并清楚理解用途
- `案件工作台` 首屏可直接看懂当前案件和关键状态
- 工作台内部存在明显的分区导航
- 原有主流程仍然可用：
  - 导入案件
  - 进入工作台
  - 查看静态简报
  - 管理 Hook 计划
  - 发起执行
  - 查看流量、证据和报告
